# 藏汉对勘状态机 · 白话操作手册

> 写给译师，不写给程序员。
> 全篇不含代码。你只需要知道：**遇到什么情况、你发什么、它回什么、卡住了怎么办。**

---

## 〇、一句话认清它是什么

**这套东西不翻译，也不判 A/B/C。它一个藏文字都不认识。**

它是**监校（纪律警察）**，不是译师。真正的翻译、分级、义理判断，全是你（或替你干活的 LLM）做的。它只做一件事——**在流程上卡你，不让你跳步、不让你把"我觉得"说成"我数过"。**

记住这句，下面全部好懂了。

---

## 一、三只手：谁干什么

一句藏文从进来到定稿，三只手依次碰它：

| 手 | 是谁 | 干什么 | 举例 |
|---|---|---|---|
| **译师** | 你 / LLM | 读藏文、给汉译、判 A/B/C、拟改译 | 「这句我判 B 级，建议改译成 X」 |
| **监校** | 状态机（这套东西） | 只做形式检查 + 卡流程，**从不碰译文对错** | 「你判了 B 级却没写改译建议，退回补」 |
| **主人** | 你（人裁） | 术语定名、引文成例、讹字取字——六类难点拍板 | 「kumārā 对音定为『孺童』」 |

**核心信条**（你自己 DEVLOG 里的话）：**不让"我觉得"冒充"我数过"。** 监校存在的唯一理由，就是防止译师图省事，把"扫了一眼觉得没问题"说成"逐句核过了"。

## 二、它凭什么能卡住你：三道硬闸门

三道闸门写死在里面，**撞上就停机报错，绝不商量**。它们只管流程动作，碰都不碰你的译文好坏。

**闸门1 · 没交初稿，不准出译文。**
你没正式提交 `[draft]`，它一个汉字都不让往外吐。
> 撞它会看到：`早停闸1：未收到 [draft] 初稿前禁止输出任何翻译内容`

**闸门2 · 对勘表没核校完，不准出分级报告。**
防止对勘表还没定稿就急着判 A/B/C。
> 撞它会看到：`早停闸2：未收到对勘表核校完成信号前，禁止输出分级报告或统稿`

**闸门3 · 磋商不可省（最狠，且层层递进）。**
哪怕你用直入模式，出统稿前也**必须**走完这一串，少一环就卡：

```
磋商做了吗？ ──否──▶ 卡：磋商环节不可省，不得径出统稿
   │是
待裁项都裁了吗？ ──否──▶ 卡：待裁项 [x] 未裁
   │是
裁了的都复诵固定了吗？ ──否──▶ 卡：裁定项 [x] 已裁但未复诵固定
   │是
你说"确认无误"了吗？ ──否──▶ 卡：待最终确认，不得出统稿
   │是
   ▼ 放行出稿
```

想抄近路？一环都省不掉。这就是你 DEVLOG 里"待裁项必经人裁"的落地。

## 三、你实际怎么"跑"它：来回发几条填表式消息

你不碰代码。就是和它来回发**结构化消息**（长得像填表的 JSON）。每一步它都把进度**整份存盘**，所以中途搁置很久、甚至换窗口，下次接着上次的点跑，不用从头来。

### 一轮典型问答长这样

```
你 ──▶  开一个切片（直入模式，pbx30a）
它 ──▶  停在【等你裁 第1条待裁项】，选项 A/B
你 ──▶  第1条选 A
它 ──▶  停在【等你最终确认】
你 ──▶  确认无误
它 ──▶  终核放行，可出统稿
```

### 你会用到的几种"信号"（照着填即可）

| 你要表达 | 你发的东西（填表式） | 什么时候用 |
|---|---|---|
| 初稿到了 | `draft: 是` | 常规入口第一步 |
| 对勘表核校完成 | `collation_confirmed: 是` | 常规入口第二步 |
| 登记一条待裁项 | `grading: [第1条，术语锁定，选项 A/B]` | 分级时发现难点 |
| 我裁定了 | `adjudications: 第1条 → A` | 它停在"等你裁"时 |
| 复诵固定 | `recite: 第1条` | 裁完之后 |
| 确认无误 | `final_confirm: 是` | 它停在"等你确认"时 |

> 具体 JSON 写法在技术文档里；你日常只需记住"表达什么→填哪一格"。

## 四、翻译活怎么接进来：你出判断，它做形式路由

**关键：你不需要让它懂藏文。你只报告"可观察的事实"，它据此形式分派。**

拿顶礼句 `ན་མོ་གུ་རུ་ཤྲཱི་བཛྲ་ཀུ་མཱ་རཱ་ཡ`（顶礼上师吉祥金刚孺童）为例：

1. **你（译师）** 察觉 `བཛྲ`(vajra)、`ཤྲཱི`(shrī)、`ཀུ་མཱ་རཱ`(kumārā) 是梵文借词，音节归属不能靠 tsheg 机械定。你报告两个**事实**：「这是梵文转写」「窗口内我数不准」。

2. **它（监校）** 做两件纯形式的事：
   - **分派**：依"是梵文转写"这个信号，归到六类里的「嵌入式咒语/种子字」。它不懂 vajra 是什么，只认信号。
   - **反模式拦截**（两头堵）：
     - 该上呈的你想蒙混 → 不行；
     - 该你自己定的、你却想甩给人裁 → 它拦你：「窗口内可自验，须自己定案，不得包装为待裁上呈」。

3. **你（人裁）** 定：「kumārā 对音取『孺童』，梵文音节按 na-mo-gu-ru-shrī-vajra-ku-mā-rā-ya 计九」，复诵固定，确认无误。它才放行。

**没有一步是它替你翻译或替你裁断。**

## 五、那"数字节"呢：它唯一敢主动做的翻译相关活

偈颂藏文音节 vs 汉字数——这个它敢自己数，因为**数数纯机械，不需要懂义理**。用的就是你 §八 那套脚本。

**但它的分寸是：只报数，绝不下结论。**

- 它会告诉你：「句3：藏 17、汉 20、差 +3、汉侧超出」
- 它**不会**说：「所以汉译错了、该删三字」——改不改、怎么改，是你的判断。
- 它甚至不敢替你定这个差是"合法逸出"还是"该修"。

> **这是 pbx30a 句3 的教训**：上一会话一度把"没逐段数就归咎藏侧"当成结论，后来被脚本逐段重数揪出其实是汉侧多配 3 字。所以现在铁律是：**数归数，结论归人。**

对梵文借词（如顶礼句），它照数不误，但会老实挂一句「机械数=X，含梵文借词、音节归属存疑，已转人裁」——绝不拿这个数去判齐不齐。

## 六、你最关心的：会不会误卡好句子？

**分两种情况，答案不同：**

### 硬闸门——不会误卡
它们只认流程动作（交没交初稿、裁没裁），**碰都不碰译文内容**。
- 你翻得再好，没走完流程也过不去；
- 你翻得再糙，只要流程走完它也不拦。
- **闸门管纪律，不管质量。** 好坏是分级报告的事。

### 洁净检测器——可能误卡，但边界收得很紧
这是八项门控里唯一真去读你**译文内容**的。它只抓三类"内部痕迹"，不该混进给读者的成稿：

| 抓什么 | 例子 |
|---|---|
| 节点编号 | `[N-13]` |
| 声部之名（在标签位） | 行首的「原颂」「论主」 |
| 操作批注 | `（批注：此句待裁）` |

**它做了精细区分，不粗暴拦：**
- 脏统稿（真带 `[N-13]`、行首「原颂：」）→ 抓出来，点名报你。
- 干净统稿里正文写「此句**引文**出自根本续，**论主**于此阐明」→「引文」「论主」是正常汉语词，**不误伤**，判为洁净。

区别在于：它只在"标签位"（行首、方括号内、后接冒号）才把这些词当声部名。正文里当普通词用，放过。

**而且**：它误判时给的是 **存疑（SUSPECT）**，不是停机——老实告诉你"这里像内部痕迹"，你看一眼就能推翻。

> 一句话：**它宁可存疑、不敢裸过；但存疑可申辩，不是终审。** 真正铁面不容商量的，只有那三道管流程的硬闸门，而它们碰都不碰你的译文。

## 七、卡住了怎么办：对症速查

| 它说 | 意思 | 你该做 |
|---|---|---|
| `早停闸1…未收到 [draft]` | 你还没交初稿 | 先正式提交初稿 |
| `早停闸2…未收到对勘表核校完成` | 对勘表没定稿 | 核校完对勘表，发"核校完成" |
| `早停闸3…磋商环节不可省` | 想跳过磋商出稿 | 老实走磋商 |
| `裁定项 [x] 已裁但未复诵固定` | 裁了但没复诵 | 发"复诵固定 第x条" |
| `待最终确认` | 就差你点头 | 发"确认无误" |
| `§五反模式…须自己定案` | 你把能自查的甩给人裁了 | 自己查证定案，别上呈 |
| `洁净FAIL…含内部痕迹` | 成稿混进了节点号/声部名/批注 | 清掉那些痕迹再出 |
| `终核缺项…未核` | 八项门控没核齐 | 补齐缺的那项证据 |

## 八、直入模式（你最常用）三句话

1. **是什么**：你带着已人工核校的对勘表，跳过前两轮（静默接收、对勘表核校），直接进第三轮分级。这是**合法变体，不是违规跳步**。
2. **省了什么**：闸门1、闸门2 视为已过。
3. **仍不能省什么**：**磋商、人裁、复诵固定、最终确认——一样都不豁免。** 闸门3 照样层层拦你。

## 九、给你的一张总图

```
        你出判断              它做形式活            你拍板
     ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
藏文─▶│ 读藏文、给汉译 │──▶│ 收结果、查格式 │──▶│  六类难点人裁  │
     │ 判A/B/C、拟改译│    │ 数音节(只报数) │    │  复诵、确认    │
     └──────────────┘    │ 卡流程三闸门   │    └──────┬───────┘
                          │ 守成稿洁净     │           │确认无误
                          └──────────────┘           ▼
                                               终核八项 ──▶ 出统稿
```

**它严的地方寸步不让（流程纪律），宽的地方尽量不误伤好句（内容洁净）。**
翻译永远是你的事，它只保证你没抄近路。

## 十、CLI 实操速查（照抄即可跑）

> 下面两串命令**都已在本地实跑验证**，每步右侧是它真实回你的信号。
> 你只需在装了 Python 的终端里，`cd` 到 `collation-statemachine` 目录，逐条粘贴。
> `--ctx` 后那个文件（如 `pbx30a.json`）是**进度存盘**——它自动记住你走到哪，换天接着跑也不丢。

### 10.1 直入模式（你最常用：带已核校对勘表直接进第三轮）

```bash
# [1] 开一个切片（直入、语体Level 2）
python run_agent.py init --ctx pbx30a.json --slice-id pbx30a --tone 2 --entry DIRECT

# [2] 推进：告诉它你分级时发现一条术语待裁（第1条，选项A/B）
python run_agent.py step --ctx pbx30a.json \
  --signal '{"grading":[{"idx":1,"atype":"TERM_LOCK","options":["A","B"]}]}'
#   → AWAIT_ADJUDICATION（停下等你裁第1条）

# [3] 你裁定：第1条选A，并复诵固定
python run_agent.py step --ctx pbx30a.json \
  --signal '{"adjudications":{"1":"A"},"recite":[1]}'
#   → AWAIT_FINAL_CONFIRM（停下等你最终确认）

# [4] 最终确认 + 交出统稿正文和对勘表数据
python run_agent.py step --ctx pbx30a.json \
  --signal '{"final_confirm":true,
             "final_text":"（你的统稿正文）",
             "table_rows":[["藏文句1","汉译1"],["藏文句2","汉译2"]]}'
#   → AWAIT_DELIVERY（终核放行，可出稿）
#   ※ 前提：认知门控七项须已登记（见下方“一个必须人工做的动作”）

# [5] 出统稿 → 门控回报
python run_agent.py step --ctx pbx30a.json
#   → AWAIT_GATE_REPORT（统稿已出，列八项状态请你只看存疑/未核）

# [6] 收讫，完成
python run_agent.py step --ctx pbx30a.json
#   → DONE
```

### 10.2 常规入口（完整四步：静默 → 对勘表 → 分级 → 统稿）

```bash
# [1] 开切片（常规入口）
python run_agent.py init --ctx pbx31.json --slice-id pbx31 --tone 2 --entry NORMAL

# [2] 推进（还没交初稿）
python run_agent.py step --ctx pbx31.json
#   → AWAIT_DRAFT（等你交 [draft] 初稿，收稿前禁翻译）

# [3] 初稿到了
python run_agent.py step --ctx pbx31.json --signal '{"draft":true}'
#   → AWAIT_COLLATION（对勘表已出、附音节表，等你核校完成）

# [4] 核校完成 + 交对勘表数据
python run_agent.py step --ctx pbx31.json \
  --signal '{"table_rows":[["藏文句1","汉译1"],["藏文句2","汉译2"]],
             "collation_confirmed":true}'
#   → AWAIT_FINAL_CONFIRM（进第三轮，无待裁则直接等确认）

# [5] 最终确认 + 统稿正文
python run_agent.py step --ctx pbx31.json \
  --signal '{"final_confirm":true,"final_text":"（你的统稿正文）"}'
#   → AWAIT_DELIVERY

# [6] 出统稿 → 门控回报
python run_agent.py step --ctx pbx31.json
#   → AWAIT_GATE_REPORT

# [7] 完成
python run_agent.py step --ctx pbx31.json
#   → DONE
```

### 10.3 信号速查（`--signal` 里能填什么）

| 你要表达 | 填这个 |
|---|---|
| 初稿到了（常规入口） | `{"draft":true}` |
| 对勘表核校完成（常规入口） | `{"collation_confirmed":true}` |
| 登记一条待裁项 | `{"grading":[{"idx":1,"atype":"TERM_LOCK","options":["A","B"]}]}` |
| 交 A/B/C 分级报告条目 | `{"graded_items":[{"grade":"A","location":"[N22]","problem":"...","tibetan_collation":"...","recommended_fix":"..."}]}` |

> **分级条目现在会当场校验结构**（不合规即打回、且不落盘）。各级必备：
> - **一律必填**：`location`（位置）、`problem`（问题）、`tibetan_collation`（藏文对勘）
> - **B 级另须**：`recommended_fix`（推荐改译）——不得只列问题
> - **C 级若出更正**：须并列 `draft_original`（初稿原文），使「初稿→更正」对照完整
> - **A 级漏译**（`is_omission:true`）：须挂 `omission_evidence`（节点比对证据）
> - **C 级禁列**：偈颂字数/节拍/逸出句/偶言句——这四类另有通道（§2.2 与 §八 脚本、排版附注、待裁第1类），不入第三轮 C 级
> - 位置可直书 `[N22]` 式节点号（对勘表自带编号径行沿用）；出稿时另有一道洁净闸把它拦在统稿之外
| 裁定 | `{"adjudications":{"1":"A"}}` |
| 复诵固定 | `{"recite":[1]}` |
| 交对勘表数据（供自动核无损覆盖） | `{"table_rows":[["藏","汉"],...]}` |
| 交统稿正文（供自动查洁净） | `{"final_text":"..."}` |
| 交偈颂逐句（供自动核音节字数） | `{"verse_pairs":[["藏文句","汉译句"],...]}` |
| 最终确认 | `{"final_confirm":true}` |

待裁六类的 `atype` 可填：`VARIANT_GRAPH`（形近异文）、`ESOTERIC_GAP`（密义存疑）、`TERM_LOCK`（术语锁定）、`SCOPE_AMBIGUITY`（辖域两可）、`QUOTE_PRECEDENT`（引文成例）、`MANTRA_BORDERLINE`（咒语归属）。

### 10.4 一个必须人工做的动作（诚实提醒）

上面 [4]/[5] 出稿前，终核要求**八项门控齐备**。其中五项属**认知判断**（声部排版、切分一贯、咒语免校、增量范围、A级全修），**信号填不了、代码也绝不替你判**——这是设计红线（认知层只接人工，绝不碰 API）。三项确定性的（无损覆盖、统稿洁净、偈颂音节字数）会在你交了 `table_rows`/`final_text`/`verse_pairs` 后**自动核**。其中偈颂音节：全齐则 PASS；有不齐句则判**存疑**并挂 tsheg 逐段物理证据（只报哪侧超出，绝不替你裁定该改哪侧），存疑会阻断出稿，须你先处置。

所以实机时，那六项认知门控需要你（或你的 handler）在出稿前**人工登记结论并挂证据**。若没登记齐，终核会如实拦你（`统稿前终核缺项…未核`）——这不是 bug，是它在履职。等你实机跑顺、决定是否要更省事的登记方式，我们再议（这也正是暂缓的丙要解决的一环）。

### 10.5 卡住了怎么读它

它每步回你一个 `signal`。`AWAIT_xxx` = 停下等你做某事（对着 10.3 填信号）；`DONE` = 走完；`error` 带 `终核…` = 门控没齐，按 10.4 补。进度全在 `--ctx` 那个 json 里，随时可查、可续。

## 十一、这项目是干嘛的：背景与预期目标

### 11.1 起点：一套跑在聊天窗口里的工作流

最早，藏汉对勘的整套规矩（`tibetan-chinese-collation/SKILL.md`）是**跑在对话窗口里**的——靠 Agent 每轮自觉遵守：先静默接收、再出对勘表、再分级、最后统稿；该数音节数音节，该上呈人裁上呈人裁。

问题在于：**规矩全靠自觉，就一定会有松动的时候。** 窗口一长、轮次一多，某一步"觉得没问题就过了"，事后很难查是哪一环松的。

### 11.2 这个项目要解决的事

把那套规矩里**确定性的那一半**，从"靠自觉"变成"代码强制"。

- 「没交初稿不准出译文」——这是死规矩，不需要判断 → **写成代码闸门**，撞上就停机。
- 「这句是不是误译」——这需要读懂藏文义理 → **留给你和 LLM**，代码碰都不碰。

一句话：**该确定的用代码焊死，该判断的留给人。**

### 11.3 预期目标（做完是什么样）

1. **本地可跑**：不依赖聊天窗口，在你自己电脑上用命令行就能驱动整套流程。
2. **不跳步**：三道硬闸门 + 八项门控，走了十轮八轮也不会悄悄漏掉某一环。
3. **有痕迹**：每一步转移、每一项门控、每一条人裁，都进审计轨迹，事后可追溯。
4. **judgment 归你**：翻译、分级、术语裁定，永远是你和 LLM 的事——代码只保证你没抄近路。
5. **绝不碰任何 API**：认知层只接**人工输入**。这是写死的红线，不是可配置项。

### 11.4 明确不做的事

- 不替你翻译，不替你判 A/B/C。
- 不替你裁术语、定成例、取底本正字。
- 不给认知判断写"检测器"——比如判"这是不是主被动颠倒"，硬做即越层，反而有害。

## 十二、原理共识：状态机 ＋ 脚本路由门控 ＋ 向 LLM 提示词注入

一句话：**这套东西不是让代码去翻译，而是让代码在每一步"把该守的规矩摆到 LLM 眼前"。**

### 12.1 四个零件各是什么

| 零件 | 大白话 | 它的本事 |
|---|---|---|
| **状态机** | "现在走到第几步"的记事本 | 不让你跳步 |
| **脚本** | 真会干活的小工具（如 §八 数音节） | 只报数，绝不下结论 |
| **门控** | 八项检查清单（过／存疑／没核） | 报"过了"却拿不出证据，当场拦下 |
| **提示词注入** | 每步把当下该守的规矩，钉进给 LLM 的任务单 | 让 LLM 不必"全靠记性" |

### 12.2 "注入提示词"到底在干嘛

想象你雇了个很厉害但健忘的译师：藏文极好，但你不能指望他每次都记得全部规矩。

办法是——**每次派活之前，先把当下这一步该守的规矩写在任务单最上头。**

> 状态机说：「现在是第三轮分级。」
> 门控说：「这一步的规矩是：B 级必须附推荐改译；C 级不许列字数问题；位置可以写 `[N22]`。」
> → 这些规矩被**拼进这一轮给 LLM 的任务单**里。

LLM 拿到的不是干巴巴一句"做分级校对"，而是一张写满当下约束的任务单。

### 12.3 为什么这个组合有效：分工 + 双保险

**分工**：
- 规矩是**确定性**的（"B级必须附改译"是死规矩）→ 交给代码，一字不差、永不疲倦。
- 翻译与义理是**认知性**的（"这句是不是误译""实操够不够典雅"）→ 交给 LLM 和你。

**双保险**：LLM 交回来的东西，代码**再查一遍结构**。B 级没附改译？当场打回，还告诉你是第几条、违反 §五 哪条。**即使 LLM 忘了任务单上的话，也漏不过去。**

### 12.4 一张图

```
        ┌─ 状态机：现在第几步？ ──┐
        │                        │
        ▼                        │
   查门控清单                     │
   "这步该守哪些规矩"              │
        │                        │
        ▼                        │
   规矩拼进任务单 ──→ 给 LLM 干活   │
                        │        │
                        ▼        │
                   交回结果       │
                        │        │
                        ▼        │
                  代码再查结构 ────┘
                  （不合规打回）
```

**转一圈：规矩进去、成果出来、再验一道。** 每轮都如此，所以走多久也不跑偏。

### 12.5 信条

> **不让"我觉得"冒充"我数过"。**

代码不懂藏文，也不假装懂。它只做两件事：**把该守的规矩，在该守的时候，摆到 LLM 眼前；再把交回来的东西，按规矩验一遍。**

## 十三、开发调试流程：实测 → 反馈 → 调优

### 13.1 这就是主循环

**不是"设计做完了、剩下打磨"，而是这类工程唯一真正有效的推进方式。**

已被反复证明——真正的进展全是这么来的，没有一次是靠"再想想"想出来的：

| 怎么发现的 | 发现了什么 |
|---|---|
| 你实机跑 | 分级报告是空的 → CLI 根本没给 A/B/C 留入口 |
| 你给出经验回放 | 旧契约会把你自己认可的理想报告整份打回 |
| 沙箱跑对照实验 | 零待裁项时，磋商被静默滑过 |
| 沙箱重数 27 句偈颂 | 对勘表 Layer A 与 Layer B 句9 汉译不一致 |

### 13.2 反馈时给什么（关键）

**给实机原始输出，别给印象。**

- ❌「感觉不太对」→ 只能猜。
- ✅ 把这些贴回来 → 能在沙箱复现同一条路径，逐字比对：
  1. 你发的**信号序列**（每步 `--signal` 填了什么）
  2. 它回的**完整 JSON**（`signal`/`state`/`detail`/`pending`，或报错全文）
  3. 那份 **`ctx.json`**（进度全量落盘，信息最全）

**凡是确定性部分对不上的，就是代码的问题**，没得辩。

### 13.3 两条循环之外的纪律

光有循环不够，容易变成"哪疼治哪、越补越乱"。所以：

**① 先写会失败的测试（红 → 绿）**
每个修复都先写一个**证明盲区真实存在**的测试，让它失败；再改代码让它转绿。这样测试就成了防回归的墙——改这里不会悄悄弄坏那里。

**② 每次反馈先分一刀：该修 vs 该留白**
有些落差不是 bug，是设计边界：
- **形式可判**的（B级有没有附改译、位置有没有露节点号）→ 补进代码。
- **认知判断**的（这是不是主被动颠倒、语体够不够典雅）→ **留白**，硬做即越层。

### 13.4 一轮调优长什么样

```
你实机跑 → 撞到落差 → 贴原始输出
    ↓
我在沙箱复现同一条路径
    ↓
分一刀：形式可判？还是认知判断？
    ↓（形式可判）
先写会失败的测试（证明盲区真实）
    ↓
改代码 → 测试转绿 → 全套件回归
    ↓
存盘、升版本号、导出干净包给你
    ↓
你再跑下一轮
```

### 13.5 每轮交付的固定形态

- 代码改动 + **对应的测试**（不写测试的修复不算完成）
- **全套件仍绿**（防回归）
- MANIFEST 升版本号、零漂移核对
- 干净包（无 `__pycache__`／`.pytest_cache`），你自行保管——**最新的包在你手里**，那是唯一的持久锚点
